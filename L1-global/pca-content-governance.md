# PCA CONTENT GOVERNANCE
## Partner Communications Architecture v1.0 — Part A
### What May and May Not Be Said

---

## What PCA Is

The Partner Communication Architecture is a cross-cutting interpretive layer that governs how structural truth becomes human language for CSPs. It is not a copywriting guide. It is not a template library. It is a constitutional constraint system.

PCA does not create rules. It translates rules. It does not modify any OS. It governs what OS may say.

**PCA has failed if any of the following are true:**
- A CSP believes something not present in structural truth
- Two teams translate the same structural state differently
- Meaning contradicts structure and is not immediately corrected

> *If meaning contradicts structure, structure prevails. Meaning must remain singular.*

---

## §1. Architectural Position

PCA is downstream of Primitives and Governance Canon — both locked. It does not require completed OS modules to function. It tests messages against structural truth that already exists.

PCA covers:
- **Part A (this file):** Content governance — what may and may not be said
- **Part B (pca-enforcement-governance.md):** Enforcement governance — how correct language is versioned, owned, audited, and enforced

## §2. Eight Constitutional Principles

Every CSP-facing message must satisfy all eight principles. Failure on any one invalidates the message. There are no partial passes.

### Principle 1: Structure First, Language Second

Every message must be traceable to a structural state, governance rule, or OS output. If no structural source exists, the message has no authority to exist.

> *Test: Can you point to the exact primitive, governance layer, or OS state this message translates? If no — delete the message.*

### Principle 2: No Persuasion, Only Information

The system does not persuade CSPs. It informs them of states, consequences, and options.

> *Test: Would this message change if the CSP were a machine? If yes — persuasion has entered.*

### Principle 3: No Hidden Economic Signaling

No message may imply income levels, suggest earning potential, frame upside narratively, or create expectations beyond the published rate card.

> *Test: Does this message create any expectation not already published in the rate card? If yes — it is economic signaling.*

### Principle 4: Role Integrity

Communication must never position the CSP as: a customer to be delighted, a partner to be persuaded, an employee to be managed, or a business owner to be empowered. The CSP executes service. The system governs outcomes.

> *Test: Does this message treat the CSP as someone whose feelings must be managed? If yes — role integrity is broken.*

### Principle 5: Impersonal Authority

All state changes must be communicated as system outcomes, not human decisions. No message may imply a person decided the consequence or that the outcome is negotiable.

> *Test: Does this message have a human subject who decided the consequence? If yes — rewrite with system as subject.*

### Principle 6: Calm Over Urgency

No message may create artificial urgency, cliff-edge anxiety, or panic about income, status, or continuation. Urgency is permitted only for FPV-level integrity breaches.

> *Test: Would a CSP reading this at 10pm feel anxious about tomorrow? If yes — the tone is wrong.*

### Principle 7: Dignity Without Indulgence

Dignity comes from clarity and predictability, not from warmth or reassurance. Respect does not mean softening structural truth.

> *Test: Is this message adding words to make the CSP feel better rather than understand better? If yes — trim.*

### Principle 8: No Alternate Interpretations

For every structural state, there must exist exactly one canonical translation. If two teams communicate the same structural state differently, the system has fractured. This principle overrides convenience, regional adaptation, and improvisation.

> *Test: If two CSPs in different cities receive different wording for the same structural state — PCA has failed.*

## §3. Vocabulary Canon

### §3.1 Forbidden Terms — Hard Ban

The following 16 terms are permanently banned from all CSP-facing communications. No exceptions for regional language, informal tone, or field context.

| Forbidden Term | Why Forbidden | Correct Replacement |
|---|---|---|
| Customer (in CSP context) | Implies ownership; triggers legacy model | Connection / Connection base |
| Lead | Implies selling required; violates P1 | Connection request |
| Your customer | Implies CSP ownership; violates P2 | Connection you service |
| Target / Goal | Push-based selling; violates belief architecture | Routing exposure / Allocation state |
| Earn more / Earning potential | Economic signaling | Rate card published / Deterministic payout |
| Penalty / Fine / Punishment | Moral framing | System consequence / State change |
| Warning | Human judgment and threat | System signal / Pattern detected |
| Demoted / Downgraded | Status language | Exposure adjusted / Eligibility changed |
| Promoted / Upgraded | Status language / entitlement | Access enabled / Eligibility expanded |
| We decided / Management decided | Human authority | System determined / State requires |
| Partner (as identity) | Implies equal governance | CSP / Connection Service Provider |
| Incentive (as motivation) | Behaviour manipulation | Deterministic payout / Published rate |
| Opportunity | Sales language | Allocation / Routing / Exposure |
| Reward / Recognize | Gamification language | Eligibility / Access |
| Trust / We trust you | Emotional manipulation | Remove entirely |
| Sorry / Apologies | Implies negotiability | System state is X / Action required is Y |

### §3.2 Canonical Terms — Mandatory Use

These 13 terms must be used whenever the concept they represent appears in a message. No paraphrasing.

| Domain | Canonical Term | Meaning |
|---|---|---|
| Work unit | Connection | Service instance attached to a location |
| New work | Connection request | System-generated demand assigned to CSP |
| Work volume | Active connections / Connection base | Total active connections serviced |
| Growth | Routing exposure | System-controlled allocation of future connections |
| Income | Base payout + Bonus (if eligible) | Deterministic economics per rate card |
| Quality state | SLA state: Compliant / At Risk / Non-Compliant | Outcome truth about user experience |
| Enforcement | System consequence | Structural response to pattern or FPV |
| Exit | System exit | Valid, dignified departure from system |
| Improvement | Enablement | Capability support — not punishment |
| Rule change | System update (versioned, dated) | Forward-effective policy evolution |
| Feedback | Voice input | Consultative signal — not negotiation |
| Internet quality | Service Stability | Uptime + speed/latency compliance |
| Device | NetBox | Wiom-owned system asset in CSP custody |

### §3.3 Hindi Vocabulary Lock

Hindi canonical terms are part of PCA — not a downstream translation exercise. They are versioned, immutable, and MGO-approved. If Hindi canonical text does not exist, the block may not be deployed in Hindi-primary channels.

| Active connections | एक्टिव कनेक्शन | आपके ग्राहक |

## §4. Tone Envelope

The allowed tone band is: **Neutral-Professional**. Not cold. Not warm.

| Dimension | Too Cold (Forbidden) | Neutral-Professional (Allowed) | Too Warm (Forbidden) |
|---|---|---|---|
| Acknowledgment | No acknowledgment | Brief, factual: "Noted." / "Received." | Effusive: "Thank you so much!" |
| State description | Bare code: "State: NC" | Clear: "Your SLA state is Non-Compliant in Service Stability domain." | Softened: "Your quality has dipped a little." |
| Next action | No guidance: "See docs." | Specific: "Enablement available. Mandatory review scheduled." | Encouraging: "We're here to help you get back on track!" |
| Greeting | None | Functional: "Hello [Name]." | Familiar: "Hi buddy!" / "Hey there!" |
| Closing | None / abrupt | Clean: "For details, see [system reference]." | Warm: "Wishing you all the best!" |
| Error / Issue | Blame: "You caused this." | Factual: "Issue detected at [location]. Steps: [X]." | Apologetic: "We're so sorry this happened." |

> *Tone Invariant: Warm enough to preserve dignity, cold enough to prevent persuasion. When in doubt, err toward factual.*

### §4.1 Hindi Tone Discipline

| Forbidden Hindi Patterns | Allowed Hindi Patterns |
|---|---|
| आप बहुत अच्छा काम कर रहे हैं! | आपका SLA स्टेट Compliant है। |
| चिंता मत करिए, हम हैं ना! | Service Stability डोमेन threshold से नीचे है। Enablement उपलब्ध है। |
| आपसे गुज़ारिश है... | सिस्टम को आपकी एक्शन की ज़रूरत है: [action] |
| भाई साहब / पार्टनर साहब | CSP [Name/ID] |

## §5. State-to-Language Translation Rules

Every structural state has one canonical translation. No improvisation. These translations form the basis for Message Blocks (§6).

### §5.1 SLA State Communication

| SLA State | Correct Communication | Forbidden Communication |
|---|---|---|
| Compliant | Your SLA state is Compliant. All domains within threshold. | Great job! Keep it up! |
| At Risk | Your SLA state has moved to At Risk. [Domain] below threshold. Enablement available. | Warning! Your performance is dropping. |
| Non-Compliant | Your SLA state is Non-Compliant. [Domain] crossed severe threshold. Routing taper active. Mandatory enablement. | You are failing. We may have to take action. |
| Recovery | SLA state returned to [state] after [N] consecutive compliant windows. | Congratulations! You've been promoted back! |

### §5.2 Routing / Allocation Communication

| Routing State | Correct Communication | Forbidden Communication |
|---|---|---|
| Full exposure | Routing: Full. Connection requests per system logic. | You're getting lots of leads! |
| Tapered | Routing exposure adjusted. Allocation reflects SLA state. | We've reduced your leads. |
| Expanded | Additional capacity available based on sustained stability. | You've been upgraded! |
| Volume fluctuation | Connection request volume varies by system demand. Normal behaviour. | Don't worry, more leads will come. |

### §5.3 Economic / Compensation Communication

| Economic State | Correct Communication | Forbidden Communication |
|---|---|---|
| Base payout | Base payout per valid recharge: [Rate card amount]. Credited automatically. | You can earn up to X per month! |
| Bonus eligible | Bonus eligibility: Active. Tier: [X] per Service Stability domain. | Keep quality high to earn more! |
| Bonus paused | Bonus eligibility paused. SLA state not Compliant. Resumes when Compliant. | You've lost your bonus. |
| Rate card change | Rate card update effective [date]. Version [X]. Forward-effective only. | Good news! Earnings increasing! |
| Carry fee | Carry fee: [amount]/day for NetBox held >15 days post-expiry. Stops on return. | You're being charged! |
| Deposit settlement | Deposit ledger: [X] issued, [Y] returned, [Z] lost. Settlement per formula. | We're deducting for lost devices. |

> *Economic Anti-Contamination: No message may contain 'earn', 'income', 'salary', 'growth' (financial), 'opportunity', or 'potential' in reference to money.*

### §5.4 Enforcement Communication

| Enforcement Type | Correct Communication | Forbidden Communication |
|---|---|---|
| Performance pattern | Sustained pattern detected in [domain]. SLA state updated. Enablement available. | We've noticed poor performance. This is your warning. |
| FPV detected | Fundamental Principle Violation: [Principle #]. System action: [action]. Integrity matter. | You have been caught violating rules. |
| Suspension | System participation temporarily suspended. Reason: [structural]. Review in progress. | You have been suspended for bad behaviour. |
| Termination | System participation terminated. Reason: [FPV type]. Settlement per process. | You have been fired. |

### §5.5 Exit Communication

| Exit Type | Correct Communication | Forbidden Communication |
|---|---|---|
| Voluntary | Exit initiated. Settlement per timeline. NetBox return per Asset Recovery OS. | Are you sure? Let's talk about what we can do. |
| System-initiated | System participation ending. Reason: [structural]. Settlement per process. | Unfortunately we have to let you go. |
| Post-exit | Final settlement completed. System access revoked. | We hope you'll come back! |

## §6. Message Block Architecture

To enforce Principle 8 (No Alternate Interpretations), every canonical translation from §5 must be registered as a versioned, immutable Message Block.

### §6.1 What Is a Message Block?

A Message Block is the only authorised language for a given structural state. It is identified by a Block ID and may not be rewritten locally.

| Element | Description |
|---|---|
| Block ID | Unique identifier: BLOCK-[OS]-[STATE]-[VERSION]. Example: BLOCK-SLA-AT-RISK-01 |
| Structural State ID | Reference to the OS state this block translates |
| Canonical Text | The exact authorised wording from §5 translations |
| Severity Classification | Informational / Attention / Action Required / Integrity / Termination |
| Channel Eligibility | Which channels may use this block |
| PCA Version | Version of PCA under which this block was approved |

### §6.2 Message Block Constraints

**Teams may:**
- Add procedural instructions
- Clarify steps
- Reference help documentation

**Teams may NOT:**
- Reframe consequences
- Soften enforcement language
- Add reassurance beyond tone envelope
- Introduce economic implication
- Paraphrase state descriptions

> *Block Rule: If a structural state has a registered Message Block, no other wording may be used for that state. Anywhere. By anyone.*

### §6.3 State-to-Block Mapping Requirement

Every CSP-facing message that communicates a structural state must declare:
- Structural State ID
- Message Block ID
- PCA Version

If no State ID exists for what is being communicated, the message may not be issued.

> *This prevents: narrative invention, policy-by-WhatsApp, and untracked language drift.*

## §7. Communication Threshold Policy

Over-communication is a form of fragmentation. Silence is a valid system signal. See comms-principles.md for the full frequency cap framework. The threshold matrix below governs channel eligibility per state.

| State Change | Dashboard | Push | Call / WhatsApp |
|---|---|---|---|
| Daily volume fluctuation | Yes (auto) | No | No |
| SLA remains Compliant | Yes (auto) | No | No |
| SLA Compliant → At Risk | Yes (auto) | Yes (once) | No |
| SLA At Risk → Non-Compliant | Yes (auto) | Yes (once) | Only if enablement requires scheduling |
| SLA recovers to Compliant | Yes (auto) | Yes (once) | No |
| Routing taper (gradual) | Yes (auto) | No | No |
| Routing significant change | Yes (auto) | Yes (once) | No |
| Bonus tier change | Yes (auto) | Yes (once) | No |
| Rate card update | Yes (auto) | Yes (advance notice) | No |
| Normal payout | Yes (wallet) | No | No |
| Carry fee accrual | Yes (wallet) | No (unless first) | No |
| FPV detected | System action | Yes (immediate) | Process only |
| Connection request assigned | Yes (auto) | Yes (action required) | No |

**Anti-Over-Communication Rules:**
- Maximum push notifications per CSP per day: 3 (excluding connection requests)
- No repeated notifications for same state within evaluation window
- No proactive check-in calls
- No motivational messages

> *Silence Rule: If no state change or action required, the message should not exist.*

## §8. Anti-Contamination Tests

Before any CSP-facing message ships, it must pass all eight tests. Failure on any one requires a rewrite — not a judgment call.

| # | Test | Question | If Failed |
|---|---|---|---|
| 1 | Structural Source | Translates a specific state / rule / OS output? | Delete. No authority. |
| 2 | Persuasion | Attempts to convince or emotionally influence? | Rewrite as factual. |
| 3 | Economic Signal | Creates expectation beyond rate card? | Remove implication. |
| 4 | Role Integrity | Treats CSP as customer / friend / subordinate? | Rewrite as agent. |
| 5 | Impersonal Authority | Human appears as decision-maker? | System as subject. |
| 6 | Calm | Would cause anxiety at 10pm? | Reduce urgency. |
| 7 | Dignity | Patronises, blames, or grades? | Neutral description. |
| 8 | Vocabulary | Contains forbidden term? | Replace with canonical. |

### §8.1 Channel-Specific Rules

| Channel | Rules |
|---|---|
| In-app notifications | Full PCA. No emoji. No exclamation marks. No motivational language. |
| Dashboard text | Full PCA. State descriptions only. No narrative. No comparison. |
| Push notifications | Full PCA. Factual. No urgency unless FPV. |
| Call centre scripts | Full PCA. Pre-approved. Must reference Block ID. No improvisation on states. |
| WhatsApp / SMS | Full PCA. Templated only from block library. No free-text economic discussion. |
| Field team verbal | PCA principles. No promises. No interpretation. Reference system state only. |
| Onboarding materials | Full PCA. Vocabulary canon enforced. No legacy framing. |

## §9. OS-Specific Communication Guardrails

### §9.1 Demand & Allocation OS
- **May:** Current allocation state, connection request volume, system demand signals
- **May NOT:** Future volume promises, area-specific forecasts, comparative allocation, reasons beyond SLA link
> *Allocation is fact. Never decision, justification, or trajectory.*

### §9.2 Quality OS / SLA Framework
- **May:** Domain-level state, metric values, thresholds, trends, evaluation window, recovery path
- **May NOT:** Comparative CSP data, ranking, percentile, moral judgment, emotional framing
> *SLA publishes truth. It does not judge, rank, or compare.*

### §9.3 Compensation OS
- **May:** Rate card amounts, bonus tier, wallet balance, carry fee, deposit ledger, settlement timing
- **May NOT:** Earning potential, projections, comparison, framing as good / bad / fair / unfair
> *Money is a rate card number. Never a narrative, promise, or judgment.*

### §9.4 Enforcement & FPV OS
- **May:** Pattern (domain, duration), consequence (specific action), FPV type, recovery path, termination reason
- **May NOT:** Character judgment, moral language, threats, warnings, negotiation openings
> *Consequences are system physics. Not opinions, threats, or negotiations.*

### §9.5 Exit OS
- **May:** Process steps, settlement timeline, asset recovery, access revocation
- **May NOT:** Retention offers, guilt, loss framing, emotional farewell, re-entry promises

### §9.6 Capability Development OS
- **May:** Enablement resources, diagnostic support, training modules, infra review
- **May NOT:** Grade capability, use patronising guidance, frame enablement as punishment

### §9.7 Partner Progression OS
- **May:** Eligibility status, changes, what to sustain for continued access
- **May NOT:** Tier names, rank, comparative standing, promotional language
> *Progression is access, never status.*

### §9.8 Voice & Change Log OS
- **May:** Feedback received, system response, version changes, rationale
- **May NOT:** Individual attribution, negotiation outcomes, promises from feedback

## §10. Escalation Tone Protocol

As severity increases, word count decreases. Never more emotion.

| Severity | Tone | Character | Example |
|---|---|---|---|
| Informational | Neutral, factual | State description. No action. | SLA state is Compliant. All domains within threshold. |
| Attention | Neutral, directive | State change. Optional action. | SLA moved to At Risk. Enablement available. |
| Action Required | Factual, clear | Consequence active. Action needed. | SLA Non-Compliant. Routing taper. Mandatory enablement. |
| Integrity | Formal, impersonal | FPV. System action. Binary. | FPV: [type]. System access suspended. |
| Termination | Formal, minimal | Ended. Process only. | Participation terminated. Settlement per process. |

> *Escalation Invariant: As severity increases, word count decreases. More serious = fewer words. Never more emotion.*

## §11. Economic Messaging Rules

### Five Hard Prohibitions

**Prohibition 1 — No Income Framing**
- Forbidden: "You can earn up to Rs 25,000/month with 80 connections."
- Correct: "Base payout: Rs 300 per valid recharge. Bonus per rate card. See wallet."

**Prohibition 2 — No Upside Narratives**
- Forbidden: "As your connections grow, your income grows too!"
- Correct: "Payout per connection per recharge cycle. Rate card published."

**Prohibition 3 — No Comparison Framing**
- Forbidden: "With traditional ISPs you earn X, with Wiom you earn Y."
- Correct: "Wiom rate card: [amount]. No collection. No churn risk."

**Prohibition 4 — No Loss Framing**
- Forbidden: "You lost your bonus because of poor quality."
- Correct: "Bonus eligibility paused. SLA state: At Risk. Resumes when Compliant."

**Prohibition 5 — No Conditional Promises**
- Forbidden: "Maintain quality and you'll definitely get more connections."
- Correct: "Routing exposure responds to SLA state. Rate card defines payout."

### §11.1 Economic Validation Checklist

Run on every message that mentions any economic element.

| # | Question | Pass |
|---|---|---|
| 1 | Every number traceable to published rate card? | Yes |
| 2 | Contains future income projection? | No |
| 3 | Uses narrative around money? | No |
| 4 | Compares economics to anything? | No |
| 5 | Uses: earn, income, salary, opportunity, potential? | No |
| 6 | Identical regardless of which CSP reads it? | Yes |
| 7 | Creates expectation not in rate card? | No |

## §12. System Update Communication Protocol

| Update Type | Advance Notice | Channel | Tone |
|---|---|---|---|
| Rate card change | 30 days | Dashboard + Push + Notice | Factual. Version-referenced. |
| Bonus formula | 30 days | Dashboard + Push + Notice | Factual. No narrative. |
| SLA threshold | 14 days | Dashboard + Push | Old vs new stated. |
| Routing logic | 14 days | Dashboard + Push | Impact structural. |
| New feature | At launch | Dashboard + Push (once) | Functional only. |
| Governance Canon | 30 days | Notice + In-app consent | Formal. |
| FPV list | 14 days | System notice | Clear on change. |

**Every update must contain exactly:**
1. What changed
2. Version reference
3. Effective date (future)
4. What does NOT change
5. Where to see details

> *Updates are version transitions. Never good news, bad news, improvements, or apologies. Never explain WHY — rationale invites negotiation.*
